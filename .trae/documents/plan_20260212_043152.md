จากการตรวจสอบระบบอย่างละเอียด พบว่าข้อมูลส่วนใหญ่เชื่อมต่อกับฐานข้อมูลจริง (Supabase) เรียบร้อยแล้ว แต่ยังมี **1 จุดสำคัญ** ที่ยังเป็นการคำนวณจำลอง (Simulation) อยู่ คือ **S-Curve (กราฟความก้าวหน้าโครงการ)**

### ผลการตรวจสอบข้อมูล (Data Audit Findings)

1.  **✅ ข้อมูลจริง (Real Data)** - เชื่อมต่อฐานข้อมูลสมบูรณ์:
    *   **Dashboard**: ตัวเลข KPI (Budget, Cost, Remaining), สถานะโครงการ, และกราฟการเงิน (Financial Overview) ดึงจาก `projects` และ `budget_lines` จริง
    *   **Projects & Tasks**: รายการโครงการ งาน และสถานะต่างๆ ดึงจากตาราง `projects` และ `tasks`
    *   **Users**: ข้อมูลผู้ใช้, Role, และ Status ดึงจากตาราง `users` (Dropdown ที่เห็นเป็น Hardcode คือค่าคงที่ Enums ที่ตรงกับ Schema)
    *   **Reports**: รายงาน Utilization และ Sunburst คำนวณจาก `time_entries` จริง
    *   **Sales Pipeline**: ดึง Deals และ Stages จากฐานข้อมูล (มีระบบสร้าง Default Stages อัตโนมัติหากยังไม่มี)

2.  **⚠️ ข้อมูลจำลอง (Simulated/Mock Data)** - ต้องปรับปรุง:
    *   **S-Curve (API `projects/[id]/scurve`)**: ปัจจุบันใช้วิธีคำนวณแบบ Linear Projection (ลากเส้นตรงจาก 0 ถึง % ปัจจุบัน) ไม่ได้ดึงจากประวัติการทำงานจริง เนื่องจากยังไม่มีระบบบันทึกประวัติความก้าวหน้า (Progress History)
    *   **API Fallbacks**: บางจุดมีการใช้ `try...catch` ที่คืนค่าว่างหรือ 0 เงียบๆ เมื่อเกิด Error ซึ่งอาจซ่อนปัญหาจริง

### แผนการปรับปรุง (Improvement Plan)

เพื่อให้ข้อมูลเป็น "Real Data" 100% ผมขอเสนอแผนการแก้ไขดังนี้:

1.  **ปรับปรุงระบบ S-Curve ให้รองรับข้อมูลจริง**:
    *   แก้ไข API `projects/[id]/scurve/route.ts` ให้พยายามดึงข้อมูลจากตาราง `project_progress_history` ก่อน
    *   สร้างกลไกบันทึก History: เมื่อมีการอัปเดต % ความก้าวหน้าของโปรเจกต์ ให้บันทึก Snapshot ลงตาราง `project_progress_history` ด้วย (เพื่อให้กราฟ S-Curve เริ่มสะท้อนความจริงตั้งแต่วันนี้เป็นต้นไป)

2.  **เพิ่มความเสถียรของ API**:
    *   ตรวจสอบจุดที่ใช้ Fallback ค่าว่าง ให้มีการ Log Error ที่ชัดเจนขึ้น เพื่อให้ทราบเมื่อข้อมูลดึงไม่ได้จริง

ท่านต้องการให้ผมดำเนินการปรับปรุง **ระบบ S-Curve** ตามแผนนี้เลยหรือไม่ครับ?
